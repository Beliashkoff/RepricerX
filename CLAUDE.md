## 1. Что это за проект

**Название:** «Сервис для автоматического управления ценами на маркетплейсах. Веб-приложение» (бренд — **RepricerX**, репозиторий — **RepricerX**).

**Суть:** веб-приложение для продавцов на маркетплейсах **Wildberries** и **Ozon**. Пользователь подключает магазины по API-ключам, импортирует свои SKU, задаёт стратегии ценообразования (следование за конкурентом, поддержание маржи, фиксированная цена), а сервис автоматически:
- собирает рыночные/конкурентные данные,
- рассчитывает рекомендуемые цены в рамках ограничений (min/max, макс. шаг изменения, маржа),
- отправляет обновлённые цены в маркетплейсы по их API,
- ведёт журнал изменений и интеграционных ошибок.

**Контекст:** курсовая работа 2 курса ОП «Программная инженерия» ФКН НИУ ВШЭ (группа БПИ244, Зуев А. И.). Оформление — по ГОСТ 19 (ЕСПД).

**Целевой результат — прототип, который:**
1. Проходит приёмку по ПМИ (см. ТЗ, раздел 8).
2. Развёртывается одной командой на Linux-сервере через `docker compose up`.
3. Подключает минимум 2 маркетплейса (WB, Ozon) и реально обновляет цены через их API.

## 3. Стек (решено)

**⚠️ В документах есть противоречие:** ТЗ п. 4.5.1.1 и оба промежуточных отчёта указывают **Python 3.10+ / FastAPI**, но ТП (Текст программы) и реальный код — **Golang + React**. Код первичнее: **работаем на Go**. Это значит, что **ТЗ и Описания 1/2 нужно будет актуализировать перед сдачей** (Python → Go, FastAPI → Gin, React оставить).

**Итоговый стек:**

| Слой | Выбор | Примечание |
|------|-------|-----------|
| Backend | **Go 1.24.x** | основной язык, версия зафиксирована в `go.mod` |
| HTTP-роутер | **Gin** (`gin-gonic/gin`) | быстрый, идиоматичный, большая экосистема middleware |
| Работа с Postgres | **pgx/v5** (чистый) | без `sqlx` и без ORM — прямые запросы, `pgxpool`, миграции в чистом SQL |
| Миграции | **golang-migrate/migrate** | CLI + библиотека; файл `000001_init.up.sql` под него |
| Планировщик | **robfig/cron v3** | для планового пересчёта цен и фоновых джоб (ТЗ 4.1.1.6.4) |
| Очередь / фон | **Redis 7** (уже в compose) + простой worker-pool на goroutines | для массовых апдейтов и ретраев |
| Frontend | **React 18 + Vite + TypeScript** | подтверждено ТП; UI-библиотека — **shadcn/ui** или **Ant Design** (таблицы + формы) |
| Инфра | **Docker + docker-compose** | Postgres 16, Redis 7, api, worker, web; **nginx** как reverse proxy |
| CI | GitHub Actions (минимум: lint + build + test) | если останется время |

**Окружение для сдачи:** Linux-сервер (VPS), доступ по SSH, домен с HTTPS (для ТЗ 4.5.4.2). Для учебной демонстрации допустим `http://<ip>:<port>` + самоподписанный TLS.

## 4. Архитектура

**ТП декларирует «микросервисы и 2 контура»** (внутренний и внешний). Для курсовой в текущих сроках полноценные отдельные сервисы с сетевым взаимодействием — избыточно. **Фактически делаем «микросервисный монолит»**: логически код разделён на независимые модули-сервисы (`auth`, `shop`, `catalog`, `market`, `strategy`, `pricing`, `dispatcher`, `audit`, `notifier`, `report`) с чёткими границами через интерфейсы, но развёртываются они как один `api`-процесс плюс `worker`-процесс. Это даёт и соответствие ТП («микросервисная разметка»), и реалистичные сроки (не надо поднимать message bus, service discovery и т. п.). При защите формулируем: **«микросервисный монолит — модульно разделённая архитектура с монолитным развёртыванием прототипа; переход к полным микросервисам — вопрос деплоя, не кода»**.

**Слоистая структура (Clean-ish Architecture):**

```
cmd/
  api/main.go              # HTTP-сервер
  worker/main.go           # планировщик + обработчик фоновых задач
internal/
  domain/                  # чистые структуры: User, Shop, SKU, Strategy, PriceChange, PricePlan
  repository/              # интерфейсы + pgx-реализации
  service/
    auth/                  # регистрация, логин, сессии (4.1.1.1)
    shop/                  # CRUD магазинов, проверка подключения (4.1.1.2)
    catalog/               # SKU: импорт, поиск, фильтры, ручное добавление (4.1.1.3)
    market/                # сбор рыночных/конкурентных данных (4.1.1.4)
    strategy/              # стратегии + ограничения (4.1.1.5)
    pricing/               # движок расчёта цен, план изменений, симуляция (4.1.1.6)
    dispatcher/            # отправка цен в МП с ретраями и rate-limit (4.1.1.7)
    audit/                 # журнал изменений цен (4.1.1.8)
    notifier/              # уведомления пользователю (4.1.1.9)
    report/                # CSV-выгрузка, агрегированные метрики (4.1.1.10)
  integration/
    wildberries/           # адаптер WB API: GetSKUs, UpdatePrice, TestAuth
    ozon/                  # адаптер Ozon API: GetSKUs, UpdatePrice, TestAuth
    marketplace.go         # общий интерфейс Marketplace
  transport/http/          # handlers, middleware (auth, rate-limit), DTO, роутинг
  scheduler/               # cron-джобы: плановый пересчёт, очистка старых логов
  config/                  # загрузка из env
  pkg/
    crypto/                # шифрование API-ключей магазинов при хранении
    httpclient/            # общий клиент с ретраями и rate-limit-ведром на магазин
    logger/                # структурные логи
migrations/
web/                       # React-приложение (Vite)
  src/
    api/                   # клиент к backend-у (tanstack-query)
    pages/                 # Вход, Регистрация, Дашборд, Магазины, SKU, Стратегии, Журнал, Настройки
    components/
docker/
  api.Dockerfile
  worker.Dockerfile
  web.Dockerfile
  nginx.conf
docker-compose.yaml
docker-compose.prod.yaml
```

**Ключевые принципы:**
- Один `Marketplace` interface — добавить Яндекс.Маркет позднее = новый адаптер без правок в движке.
- API-ключи магазинов в БД **только в зашифрованном виде** (AES-GCM, ключ из env). ТЗ 4.5.4.1.
- Идемпотентность апдейтов цен: `correlation_id` в каждом запросе + дедуп по (sku_id, target_price) в окне.
- Все длинные операции (импорт, массовый пересчёт, массовая отправка) — не в HTTP-хендлере, а в воркере; UI опрашивает статус джобы.

## 5. Модель данных

За основу берётся существующая миграция (`migrations/000001_init.up.sql`). **Её нужно доработать** — в текущем виде не хватает сущностей из ТЗ:

### Есть
`users`, auth-таблицы `sessions`/`email_verifications`, унифицированные `shops`, `products`, `competitors`, `strategies`, `strategy_assignments`, `price_plans`, `price_plan_items`, `price_change_log`, `integration_log`, `import_log`.

### Проблемы и правки
1. **Таблицы WB/Ozon унифицированы миграцией `000003_unify_marketplaces`**:
   - `shops(id, user_id, marketplace enum('wb','ozon','ym'), name, credentials_encrypted, status, last_checked_at, created_at)`
   - `products(id, shop_id, external_sku, name, current_price, currency, status, min_price, max_price, cost_price, updated_at, UNIQUE(shop_id, external_sku))`
   - `competitors(id, product_id, external_sku, url, price, source, fetched_at)` + опционально `competitors_history` либо JSONB поле.
2. **`payments`** — не из ТЗ, не нужно для приёмки. Удаляется в `000003_unify_marketplaces`.
3. **Таблицы, требуемые ТЗ, добавлены на уровне БД; следующий шаг — доменные модели, репозитории и сервисы:**
   - `strategies(id, user_id, name, type, params jsonb, constraints jsonb, fallback_policy, priority, enabled, created_at)` — типы: `below_median_pct`, `min_competitor_plus_step`, `min_margin_pct`, `fixed`.
   - `strategy_assignments(id, strategy_id, product_id nullable, filter jsonb nullable, priority)` — явное vs по фильтру.
   - `price_plans(id, shop_id, status, created_at)` и `price_plan_items(id, plan_id, product_id, current_price, target_price, final_price, constraint_hit, status, error)`.
   - `price_change_log(id, shop_id, product_id, strategy_id, old_price, new_price, target_price, reason, constraint_hit, status, correlation_id, created_at)` — журнал по ТЗ 4.1.1.8.1, **хранение ≥ 180 суток**.
   - `integration_log(id, shop_id, operation, http_status, error_text, correlation_id, created_at)` — ТЗ 4.1.1.12.3, **хранение ≥ 30 суток**.
   - `import_log(id, shop_id, started_at, finished_at, added, updated, errors)` — ТЗ 4.1.1.12.2.
   - `sessions(id, user_id, token_hash, expires_at)` — TTL 24 ч (ТЗ 4.1.1.1.2.4).
4. **В `users`** поля auth (`password_hash`, `display_name`, `status`, lockout) добавлены миграцией `000002_auth`.
5. **Индексы:** базовые индексы для `products`, `competitors`, `price_change_log`, `integration_log`, `import_log` добавлены.

## 8. Ключевые решения и TBD

**Принятые решения (фиксируем в проекте):**
- Язык backend — **Go** (Gin + pgx/v5, чистый), frontend — **React 18 + Vite + TS**.
- Архитектура — **микросервисный монолит** с `cmd/api` и `cmd/worker`, модули с чёткими границами через интерфейсы.
- БД — **PostgreSQL 16**, миграции — `golang-migrate` на чистом SQL, без ORM.
- Планировщик — **robfig/cron v3**; очередь/кэш — **Redis 7**.
- Reverse proxy — **nginx**.
- Валюта в прототипе — только **RUB**.
- Rate-limit по умолчанию — **5 rps на магазин**.
- TTL сессии — **24 ч** неактивности.

**Открытые TBD (решить в процессе, не блокируют старт):**
- Конкретный UI-kit: shadcn/ui vs Ant Design (рекомендация: shadcn для меньшей зависимости, AntD — быстрее собрать таблицы).
- Способ HTTPS: Let's Encrypt через certbot vs self-signed для демо-стенда.
- Источник «индикатора спроса» — в MVP не обязателен, можно не реализовывать.
- Источник конкурентных цен — в MVP берём из публичного API WB/Ozon (поиск по похожим товарам) или оставляем ручной ввод. Парсинг/скрапинг не делаем — вне скоупа и юридически сомнительно.

## 9. Конвенции разработки

Из `README.md`:
- Один коммит = одна фича. Типы: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`.
- Формат: `<тип>(<область>): <короткое описание>`.
- Один PR = одна фича. Draft PR, self-review.

**Дополнительно (рекомендация):**
- Go: `golangci-lint` в CI, стандартное форматирование `gofmt -s`.
- Frontend: ESLint + Prettier.
- Ветки: `main` — стабильная; работаем в feature-ветках → PR → `main`.
- Не комментировать ради комментариев. Комментарий нужен, когда объясняет *почему*, а не *что*.
